# ACLS Training 초기 기획

## 배경

응급의학과 의사 손동근과의 논의에서 시작. 2차병원/요양병원 간호사들도 CPR·ACLS 숙련도가 낮은 경우가 많다는 문제의식에서 출발.

`training-simulator`가 다루는 범위는 BLS(Basic Life Support, 심정지 인지·고품질 CPR·조기 제세동 중심)이고, ACLS(Advanced Cardiovascular Life Support)는 여기에 리듬 판독, 약물, 전문기도, 가역적 원인 처치와 팀 기반 소생술이 추가된다. ACLS는 분기와 반복이 많아 같은 방식의 선형 스크린 흐름으로 담기 어렵다.

## BLS vs ACLS

| 구분 | BLS | ACLS |
| ---- | --- | ---- |
| 주체 | 일반인 및 의료진 | 훈련된 의료 전문가 |
| 흐름 | 선형 (압박/환기 반복) | 분기형 (리듬 판단에 따라 경로가 갈림) |
| 현재 구현 | `training-simulator`에서 다룸 | 미구현, 이 기획의 대상 |

## ACLS 핵심 분기 구조

1. **리듬 판단이 첫 갈림길**
   - Shockable: 심실세동(VF) / 무맥성 심실빈맥(pulseless VT)
   - Non-shockable: 무맥성 전기활동(PEA) / 무수축(Asystole)

2. **Shockable 경로**
   제세동 → 2분 CPR → 리듬 재확인 → 에피네프린(3~5분 간격) → 항부정맥제 → 반복

3. **Non-shockable 경로**
   제세동 없이 즉시 CPR + 에피네프린 → 가역적 원인 감별(H's and T's 10개 항목)

4. **병원 전 → 병원 내 인계**
   인계는 핵심 심정지 알고리즘의 리듬 분기가 아닌 별도 팀 커뮤니케이션 시나리오다. 첫 MVP 범위에서는 제외한다.

## 설계 방향 메모

- 리듬 분기(Shockable vs Non-shockable)를 최상위 분기점으로 두고, 각 경로별로 별도 플로우 구성
- 단일 체크리스트/선형 스크린(`training-simulator`의 Intro→Preview→Training→Result 구조)으로는 표현 불가 — 반복, 경로 전환 및 독립 타이머를 다룰 수 있는 순환형 상태 모델 필요
- 아직 UI/기능 설계 전 단계, 방향성만 기록

## MVP 범위 결정

> 구현 기준 시나리오는 [`SCENARIO_VF_01.md`](./SCENARIO_VF_01.md)를 따른다. 현재 MVP는 전체 ACLS 알고리즘이 아니라 **일반 병동 VF 심정지의 발견부터 첫 shock 직후 CPR 재개까지**를 완성도 있게 재현하는 vertical slice다.

### 대상 사용자

- 신규 간호사
- 일반 병동 간호사
- 응급실 간호사

경력과 근무 부서에 따라 숙련도 차이가 클 수 있으므로, MVP에서는 동일한 시나리오를 제공하되 결과 화면에서 행동별 근거를 함께 설명한다.

### 학습 목표

- 2025 AHA 성인 심정지 알고리즘의 순서와 핵심 분기 암기
- 제한시간이 있는 상황에서 처치를 직접 선택하며 알고리즘을 간접 체험
- 실제 술기 숙련이나 임상적 의사결정 능력의 인증은 MVP 범위에서 제외

### 조작 및 진행 방식

- 학습자가 제한시간 안에 다음 처치를 직접 선택하는 시뮬레이션
- 오답이나 처치 지연이 발생해도 즉시 종료하지 않고 시나리오를 계속 진행
- 행동과 시간을 이벤트 로그로 기록하고 종료 후 디브리핑에서 피드백 제공

### MVP 시나리오

- 첫 MVP는 **일반 병동에서 발생한 성인 VF 심정지 고정 사례 1개**만 구현
- 시작은 병동 모니터 알람과 환자 발견, 종료는 첫 shock 직후 CPR 재개로 한정
- 시나리오 전체 길이는 사용자 지연을 포함해 최대 90초
- 약물, 반복 제세동, PEA/Asystole, 병원 전→병원 내 인계, 소생 후 처치, 서맥·빈맥 알고리즘은 제외

VF를 선정한 이유는 최초 반응, 코드 호출, CPR, 리듬 판독, 안전 확인, 제세동과 즉시 CPR 재개라는 초기 ACLS의 핵심 행동을 하나의 밀도 높은 장면에서 경험할 수 있기 때문이다.

### 채점 기준

| 기준 | 평가 내용 |
| --- | --- |
| 정확성 | 현재 상태에 적절한 처치를 선택했는가 |
| 순서 | 환자 확인, 코드 호출, CPR, 리듬 확인과 제세동 순서를 지켰는가 |
| 제한시간 | 각 의사결정을 허용 시간 안에 수행했는가 |
| 불필요한 처치 | 현재 상태에 필요하지 않거나 금기인 처치를 선택했는가 |
| CPR 중단시간 | 리듬 확인과 제세동 전후의 압박 중단을 최소화했는가 |

점수만 표시하지 않고 각 행동의 선택 시점, 정답 여부, 권장 행동 및 근거를 타임라인으로 제공한다. 세부 배점과 허용 시간은 시나리오 작성 및 의료 검수 단계에서 확정한다.

### 종료 조건

MVP에서는 임상적인 소생 노력 중단 또는 ROSC 판단을 학습자에게 요구하지 않는다. 다음 두 조건으로 시뮬레이션을 종료한다.

1. **정상 완료:** 첫 shock 후 5초 안에 CPR 재개
2. **제한 완료:** shock 후 10초까지 CPR이 재개되지 않거나 시나리오 전체가 90초에 도달

오답, 처치 누락, 제한시간 초과는 원칙적으로 즉시 종료 조건이 아니라 재촉 보고, NPC 개입, 감점 및 디브리핑 항목으로 처리한다. 실제 환자의 ROSC 또는 소생 중단을 모사하는 기능은 후속 시나리오로 둔다.

### 콘텐츠 및 검수 기준

- 콘텐츠 버전: **2025 American Heart Association Guidelines for CPR and ECC 기반**
- 알고리즘, 약물 용량, 처치 순서 및 피드백 문구에 기준 버전을 표시
- 교육용 시뮬레이션이며 실제 진료 판단이나 공식 ACLS 자격 인증을 대체하지 않음을 명시
- 초기 기획과 프로토타입은 먼저 진행하되, 배포 또는 실제 교육 적용 전 의료 전문가 검수 필요

## 병동 시각화 시뮬레이션 기획

### 제품 경험 정의

이 제품은 알고리즘을 문답식으로 푸는 퀴즈가 아니라, **병동의 코드 블루 상황 안에서 학습자가 처치를 지시하고 그 결과를 관찰하는 시각적 시뮬레이션**이다. 알고리즘과 채점 엔진은 화면 뒤에서 동작하며, 학습자는 실제 병실의 환자·의료진·장비를 통해 상황을 이해해야 한다.

학습자가 느껴야 하는 핵심 경험은 다음과 같다.

1. 환자 상태와 모니터를 관찰한다.
2. 제한시간 안에 필요한 처치를 선택하거나 팀원에게 지시한다.
3. 여러 처치가 병행되는 동안 환자, 의료진, 장비의 변화를 본다.
4. 선택과 지연이 생체신호와 다음 상황에 영향을 주는 것을 경험한다.
5. 종료 후 실제 수행 과정을 타임라인으로 복기한다.

### MVP 시각화 방식

- **고정 사선 구도의 인터랙티브 2D/2.5D 병실**을 MVP 방향으로 확정한다.
- 병실 배경 위에 환자, 의료진, 장비, 이펙트를 각각 독립적인 2D 레이어로 배치한다.
- 환자·모니터·제세동기·호출 장치 등 핵심 영역은 클릭하거나 선택할 수 있으며, 필요할 때 2D 확대 패널을 연다.
- 환자 확인, CPR, 패드 부착, 제세동과 환기는 상태별 이미지 전환이나 짧은 스프라이트/모션 애니메이션으로 표현한다.
- 원근감, 객체 겹침, 그림자와 이동 모션을 사용해 공간감을 주되 3D 모델, 물리 엔진과 자유 카메라 조작은 사용하지 않는다.
- 사실적인 그래픽보다 처치 상태와 장비 사용 여부가 즉시 구분되는 명확한 시각 피드백을 우선한다.
- 병실 장면과 시뮬레이션 로직을 분리하여 시각 자산을 교체해도 채점과 환자 상태 엔진에 영향을 주지 않게 한다.

#### 2D 레이어 구성

```text
UI 오버레이       타이머 · 작업 상태 · 처치 메뉴 · 이벤트 로그
이펙트 레이어     선택 강조 · 충전 · shock · 알람
전경 레이어       이동 중인 의료진 · 제세동기/기도 장비
환자/처치 레이어  환자 · CPR · 환기 · 패드
장비 레이어       모니터 · 제세동기 · 호출 장치 · 산소
배경 레이어       병실 · 병상 · 벽 · 바닥
```

DOM/CSS와 SVG 또는 투명 배경 이미지로 우선 구현하고, 복잡한 연속 애니메이션이 필요한 일부 요소에만 Canvas 사용을 검토한다.

### 병실 장면 구성

| 영역/객체 | 시각적 역할 | 주요 상태 |
| --- | --- | --- |
| 환자 및 병상 | 모든 처치의 중심 | 무반응, 상태 확인, CPR 중, 제세동 직전·직후 |
| 환자 모니터 | ECG와 주요 생체신호 표시 | 알람, 연결 중, VF, shock 직후 |
| 제세동기 | 패드 부착, 충전, shock 수행 | 미연결, 연결, 충전, shock 가능, 방전 |
| 산소·앰부백 | 환기 처치 표현 | 미사용, 준비, 환기 중 |
| 호출 장치 | 도움 요청과 코드 블루 활성화 | 대기, 호출, 팀 이동 중, 팀 도착 |
| 의료진 | 역할 분담과 동시 행동 표현 | 대기, 이동, CPR, 기도, 제세동 |
| 시계/코드 타이머 | 제한시간 전달 | 전체 경과시간, 장면별 지연시간 |

첫 MVP 병실에는 학습에 직접 필요하지 않은 가구나 장비를 최소화한다. 클릭 가능한 객체는 윤곽선, 포커스 또는 상황별 강조 표시로 구분하되 정답을 직접 노출하지 않는다.

### 팀 구성과 사용자 역할

MVP에서 학습자는 **코드 팀 리더 역할**을 맡아 처치를 선택하고 팀원에게 지시한다. 개별 술기를 마우스로 정밀 수행하는 방식은 기존 CPR 시뮬레이터와 분리한다.

| 역할 | 시뮬레이션 행동 |
| --- | --- |
| 압박 담당 간호사 | CPR 시작·중단·교대 |
| 기도 담당 | 산소 공급, 앰부 환기, 전문기도 준비 |
| 제세동 담당 | 패드 부착, 리듬 확인, 충전 및 shock |
| 기록 담당 | 처치 시각과 CPR 주기 기록 |

첫 MVP에서는 3~4명의 팀원을 화면에 표현하되, 학습자가 역할마다 직접 이동시키지 않는다. 처치 명령을 내리면 해당 역할의 팀원이 대상 장비로 이동하여 준비와 수행 애니메이션을 실행한다.

### 상호작용 방식

1. 병실의 환자, 모니터, 제세동기 또는 팀원을 선택한다.
2. 선택한 대상에 가능한 행동 메뉴가 문맥에 맞게 열린다.
3. 처치를 선택하면 준비 시간이 시작되고 담당 의료진이 행동한다.
4. shock처럼 안전 확인이 필요한 행동은 `충전 → 모두 물러남 확인 → shock` 단계를 분리한다.
5. 수행 결과는 장면 변화, 모니터 변화, 팀 음성 및 짧은 상태 메시지로 전달한다.

행동 메뉴는 가능한 선택지를 제공하되 권장 행동을 색상으로 미리 알려주지 않는다. 명백히 불가능한 행동만 비활성화하고, 시점상 부적절하거나 불필요한 처치는 선택할 수 있게 하여 채점에 반영한다.

### 시간과 동시 행동

실제 코드 상황처럼 CPR, 기도 준비와 제세동기 연결은 동시에 진행될 수 있어야 한다.

- 시뮬레이션 시간과 화면 애니메이션 시간을 분리해 관리한다.
- 각 처치는 `요청 → 준비 → 수행 → 완료/취소` 상태를 갖는다.
- CPR은 별도 지속 작업으로 관리하며 리듬 확인과 shock 시 일시 중단된다.
- CPR이 중단된 모든 구간을 누적하여 중단시간 채점에 사용한다.
- 첫 압박 시작, 모든 압박 중단과 재개, 리듬 확인, 충전, shock 시각을 독립적으로 기록한다.
- 학습자가 판단할 수 있도록 패드 연결, 충전과 안전 확인 완료를 팀 음성으로 알린다.

### 시각·음향 피드백 원칙

- ECG 리듬과 수치 변화는 환자 상태 엔진의 결과로만 변경한다.
- shock 순간, 흉부압박과 앰부 환기는 짧고 명확한 애니메이션으로 표현한다.
- 모니터 알람, 제세동기 충전음, 팀원의 구두 보고, 코드 타이머를 사용해 긴박감을 만든다.
- 정보 전달을 소리에만 의존하지 않도록 모든 음성 보고에 자막 또는 텍스트 로그를 제공한다.
- 교육 중에는 정답/오답을 즉시 표시하지 않고 임상적으로 관찰 가능한 결과만 보여준다.
- 개발·연습 모드에서만 선택적으로 힌트를 제공한다.

### 첫 VF 시나리오 장면 흐름

| 장면 | 상황 | 학습자의 핵심 행동 | 시각적 변화 |
| --- | --- | --- | --- |
| 알람 | 야간 병동 모니터 알람 | 해당 병상 확인 및 병실 진입 | 알람 점멸, 무반응 환자 노출 |
| 환자 확인 | 환자 무반응 | 반응·호흡·맥박 확인 | 어깨 자극, 호흡·경동맥 확인 |
| 초기 대응 | 심정지 확인 | 코드 호출과 CPR 시작 | 코드 방송, 팀 접근, 압박 시작 |
| 팀 도착 | 제세동기와 팀 진입 | CPR 유지, 패드 연결과 산소 준비 | 역할별 의료진 이동과 동시 작업 |
| 리듬 확인 | 모니터 연결 | 압박 중단 최소화, VF 판독, 충전 | VF 확대, 충전음과 장비 상태 변화 |
| 제세동 | 충전 완료 | 안전 확인, shock, 즉시 CPR 재개 | 팀 물러남, shock, 압박 재개 |
| 종료 | 첫 shock 직후 CPR 진행 | 수행 결과 확인 | 장면 정지 후 디브리핑 전환 |

시나리오의 환자 반응은 완전한 자유 시뮬레이션이 아니라 교육 목표에 맞춘 결정론적 진행으로 구성한다. 핵심 처치를 놓쳐도 시나리오는 가능한 범위에서 계속 진행하고, 최종 결과와 디브리핑에서 영향을 설명한다.

├──────────────────────────────────────────┴─────────────────┤

화면을 알고리즘 안내 패널로 채우지 않는다. 필요한 정보는 병실 객체와 모니터에서 먼저 읽게 하고, 우측 패널은 현재 진행 중인 팀 작업과 완료 보고를 확인하는 보조 수단으로 사용한다.

### 시뮬레이션과 채점의 연결

모든 사용자 명령과 시스템 결과를 하나의 이벤트 로그로 남긴다.

```ts
type SimulationEvent = {
  id: string;
  simulationTimeMs: number;
  actor: "learner" | "team" | "patient" | "system";
  action: string;
  target?: string;
  status: "requested" | "started" | "completed" | "cancelled";
  metadata?: Record<string, unknown>;
};
```

장면 애니메이션, 환자 상태 변화, 타이머와 채점은 모두 이 이벤트를 기준으로 동기화한다. 결과 화면에서는 동일한 로그를 이용해 실제 수행과 2025 AHA 기반 권장 흐름을 나란히 비교한다.

### MVP에서 제외할 시각화

- 병동 전체를 자유롭게 돌아다니는 1인칭 이동
- 3D 모델, 물리 엔진 및 자유 카메라 조작
- 여러 병실 또는 응급실 간 이동
- 팀원과의 자유 대화 또는 음성 인식
- 정밀한 손동작과 약물 용량 계산 UI
- 모든 H's and T's 처치 애니메이션
- 환자 외형의 세밀한 생리 변화나 물리 기반 시뮬레이션
- 다인 협동 플레이

### MVP 완료 기준

- 한 개의 VF 시나리오가 병동 알람부터 첫 shock 직후 CPR 재개 또는 제한 종료까지 60~90초 안에 진행된다.
- 환자, 모니터, 호출 장치, 제세동기, 산소/앰부백과 역할별 의료진이 한 병실에 표현된다.
- 환자 확인, 코드 호출, CPR, 패드 연결, 리듬 판독, 충전, 안전 확인, 제세동 및 CPR 재개를 지시할 수 있다.
- 최소 2개 이상의 처치가 동시에 진행될 수 있다.
- ECG, 장비 상태, 팀 애니메이션과 이벤트 로그가 동일한 시뮬레이션 상태를 반영한다.
- CPR 중단시간과 처치 선택·순서·지연·불필요한 행동이 결과에 집계된다.
- 사운드를 끈 상태에서도 자막과 시각 정보만으로 진행할 수 있다.
- 재시작하면 모든 시나리오 상태와 타이머가 동일한 초기값으로 복원된다.

## 2026-07-06 세션 논의 — 분기 구조 상세

**당시 상태:** 진행 여부와 스코프를 검토하던 기록. 현재 결정 사항은 위의 `MVP 범위 결정`을 따른다.

"ACLS가 스코프를 줄일 정도로 복잡한가?"라는 질문에서 시작해 분기 구조를 자세히 뜯어봄.

### 전체 트리

```
심정지 인지 → CPR 시작 + 모니터/제세동기 부착
            │
            ▼
      리듬 확인 (2분마다 반복)
            │
   ┌────────┴────────┐
Shockable          Non-shockable
(VF / 무맥성 VT)    (PEA / 무수축)
   │                  │
   ▼                  ▼
 제세동           제세동 없이 CPR 계속
   │                  │
   ▼                  ▼
CPR 2분          에피네프린 즉시 투여
   │                  │
   ▼                  ▼
리듬 재확인 ──┐   H's and T's 가역적 원인 감별
   │         │        │
(반복)   더 이상        ▼
         shockable   원인별 처치 후 CPR 계속
         아니면→ 반대 분기로 전환
```

### 복잡도가 발생하는 4개 지점

1. **리듬 판단 (2way, 그러나 고정 아님)**
   Shockable ↔ Non-shockable, 2분마다 재평가하면서 중간에 경로가 바뀔 수 있음 (예: PEA로 시작 → 리듬이 VF로 변하면 shockable 분기로 전환)

2. **약물 타이밍 (조건이 시간 기반)**
   에피네프린 3~5분 간격 반복. Shockable 경로의 항부정맥제(아미오다론/리도카인)는 "몇 번째 제세동 이후"인지에 따라 투입 시점이 달라짐 — 단순 if/else가 아니라 타이머 조건

3. **H's and T's (10개 가역적 원인 항목)** — 원인별 미니 서브플로우
   | 원인 | 처치 |
   | --- | --- |
   | Hypovolemia (혈량 부족) | 수액 |
   | Hypoxia (저산소증) | 기도/산소 |
   | Hydrogen ion, 산증 | 환기/중탄산염 |
   | Hypo/Hyperkalemia (전해질 이상) | 전해질 교정 |
   | Hypothermia (저체온) | 재가온 |
   | Tension pneumothorax (긴장성 기흉) | 바늘 감압 |
   | Tamponade (심낭압전) | 심낭천자 |
   | Toxins (중독) | 해독제 |
   | Thrombosis, pulmonary (폐혈전증) | 혈전용해/시술 |
   | Thrombosis, coronary (관상동맥 혈전증) | 관상동맥 중재술 등 |

   각 원인마다 "의심 근거 → 확인 방법 → 처치"가 전부 달라 개별 플로우가 필요.

4. **탈출 분기**
   자발순환 회복(ROSC) 시 이 알고리즘을 벗어나 완전히 다른 알고리즘(소생 후 처치)으로 전환. 반대로 소생 노력 중단(termination) 판단도 별도 분기로 존재.

### 결론 (엔지니어링 관점)

상태 개수 자체는 크지 않지만, 2분 CPR 주기, 리듬 경로 전환, 약물 타이머와 이벤트 로그를 함께 관리해야 한다. 프로토타입부터 임상 상태와 화면 상태를 분리하고 reducer 또는 명시적인 상태 머신을 사용한다. 진짜 복잡도는:
- **콘텐츠 정확성**: H's and T's 10개 항목 + 약물 타이밍 등 의학적으로 정확한 기준을 다 채워야 함
- **채점 깊이**: BLS처럼 각 분기점마다 실시간 정답/오답 평가(게이지 수준)까지 하려면 분기점 수만큼 평가 기준 정의가 필요 → 작업량이 곱해짐

"ACLS" 전체(서맥/빈맥 알고리즘, ROSC 후 처치 등 포함)로 보면 스코프가 너무 넓지만, **성인 심정지 고정 사례 1개**로 좁힌 지금 범위는 적절한 MVP로 판단한다. 학습자가 제한시간 안에 처치를 직접 선택하고, 정확성·순서·시간·불필요한 처치·CPR 중단시간을 이벤트 로그 기반으로 평가한다.
