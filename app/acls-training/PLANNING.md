# ACLS Training 초기 기획

## 배경

응급의학과 의사 손동근과의 논의에서 시작. 2차병원/요양병원 간호사들도 CPR·ACLS 숙련도가 낮은 경우가 많다는 문제의식에서 출발.

`training-simulator`가 다루는 범위는 BLS(Basic Life Support, 현장 기본 소생술)뿐이고, ACLS(Advanced Cardiovascular Life Support, 병원 전문 소생술)는 프로토콜 자체가 다르며 분기가 많아 같은 방식(선형 스크린 흐름)으로 담기 어렵다.

## BLS vs ACLS

| 구분 | BLS | ACLS |
| ---- | --- | ---- |
| 주체 | 현장의 누구나 | 병원 내 의료진 |
| 흐름 | 선형 (압박/환기 반복) | 분기형 (리듬 판단에 따라 경로가 갈림) |
| 현재 구현 | `training-simulator`에서 다룸 | 미구현, 이 기획의 대상 |

## ACLS 핵심 분기 구조

1. **리듬 판단이 첫 갈림길**
   - Shockable: 심실세동(VF) / 무맥성 심실빈맥(pulseless VT)
   - Non-shockable: 무맥성 전기활동(PEA) / 무수축(Asystole)

2. **Shockable 경로**
   제세동 → 2분 CPR → 리듬 재확인 → 에피네프린(3~5분 간격) → 항부정맥제 → 반복

3. **Non-shockable 경로**
   제세동 없이 즉시 CPR + 에피네프린 → 원인 감별("H's and T's" 8가지)

4. **병원 전 → 병원 내 인계**
   Out-of-hospital(현장) 처치에서 병원 내 ACLS로 넘어가는 인계 시점 판단까지 얽혀 있음

## 설계 방향 메모

- 리듬 분기(Shockable vs Non-shockable)를 최상위 분기점으로 두고, 각 경로별로 별도 플로우 구성
- 단일 체크리스트/선형 스크린(`training-simulator`의 Intro→Preview→Training→Result 구조)으로는 표현 불가 — 트리형 상태 모델 필요
- 아직 UI/기능 설계 전 단계, 방향성만 기록

## 2026-07-06 세션 논의 — 분기 구조 상세

**현재 상태: 진행 여부/스코프 고민 중 (결정 안 됨)**

"ACLS가 스코프를 줄일 정도로 복잡한가?"라는 질문에서 시작해 분기 구조를 자세히 뜯어봄.

### 전체 트리

```
심정지 인지 → CPR 시작 + 모니터/제세동기 부착
            │
            ▼
      리듬 확인 (2분마다 반복)
            │
   ┌────────┴────────┐
Shockable          Non-shockable
(VF / 무맥성 VT)    (PEA / 무수축)
   │                  │
   ▼                  ▼
 제세동           제세동 없이 CPR 계속
   │                  │
   ▼                  ▼
CPR 2분          에피네프린 즉시 투여
   │                  │
   ▼                  ▼
리듬 재확인 ──┐   H's and T's 원인 감별 (8갈래)
   │         │        │
(반복)   더 이상        ▼
         shockable   원인별 처치 후 CPR 계속
         아니면→ 반대 분기로 전환
```

### 복잡도가 발생하는 4개 지점

1. **리듬 판단 (2way, 그러나 고정 아님)**
   Shockable ↔ Non-shockable, 2분마다 재평가하면서 중간에 경로가 바뀔 수 있음 (예: PEA로 시작 → 리듬이 VF로 변하면 shockable 분기로 전환)

2. **약물 타이밍 (조건이 시간 기반)**
   에피네프린 3~5분 간격 반복. Shockable 경로의 항부정맥제(아미오다론/리도카인)는 "몇 번째 제세동 이후"인지에 따라 투입 시점이 달라짐 — 단순 if/else가 아니라 타이머 조건

3. **H's and T's (8가지 원인 감별)** — 사실상 8개의 미니 서브플로우
   | 원인 | 처치 |
   | --- | --- |
   | Hypovolemia (혈량 부족) | 수액 |
   | Hypoxia (저산소증) | 기도/산소 |
   | Hydrogen ion, 산증 | 환기/중탄산염 |
   | Hypo/Hyperkalemia (전해질 이상) | 전해질 교정 |
   | Hypothermia (저체온) | 재가온 |
   | Tension pneumothorax (긴장성 기흉) | 바늘 감압 |
   | Tamponade (심낭압전) | 심낭천자 |
   | Toxins (중독) | 해독제 |
   | Thrombosis (관상동맥/폐색전) | 혈전용해/시술 |

   각 원인마다 "의심 근거 → 확인 방법 → 처치"가 전부 달라 개별 플로우가 필요.

4. **탈출 분기**
   자발순환 회복(ROSC) 시 이 알고리즘을 벗어나 완전히 다른 알고리즘(소생 후 처치)으로 전환. 반대로 소생 노력 중단(termination) 판단도 별도 분기로 존재.

### 결론 (엔지니어링 관점)

상태 개수 자체(10여 개)는 크지 않음 — `training-simulator`의 `screenState` 패턴(useState + 분기 렌더링)을 확장하면 구조적으로는 충분히 감당 가능. 진짜 복잡도는:
- **콘텐츠 정확성**: H's and T's 8개 항목 + 약물 타이밍 등 의학적으로 정확한 기준을 다 채워야 함
- **채점 깊이**: BLS처럼 각 분기점마다 실시간 정답/오답 평가(게이지 수준)까지 하려면 분기점 수만큼 평가 기준 정의가 필요 → 작업량이 곱해짐

"ACLS" 전체(서맥/빈맥 알고리즘, ROSC 후 처치 등 포함)로 보면 스코프가 너무 넓지만, **심정지 알고리즘 하나**로 좁힌 지금 범위는 이미 적절한 축소로 판단됨. 다만 채점 깊이(단순 분기 선택 vs 실시간 정밀 평가)는 아직 미정 — 다음 논의에서 결정 필요.
